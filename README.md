# JuniorFlow AI

JuniorFlow AI is a first-job simulator for aspiring junior developers, built for the Education category of OpenAI Build Week 2026.

Users configure their experience and technology stack, receive a realistic engineering ticket generated by GPT-5.6, submit an approach and code or pseudocode, and receive an educational review from a simulated senior developer.

## Features

- Professional responsive landing page and guided simulation setup
- Realistic tickets generated with the OpenAI Responses API and GPT-5.6
- Structured Outputs validated with shared Zod schemas
- Educational solution reviews with scores, bugs, security concerns, and study recommendations
- Versioned browser history stored in localStorage
- Search, status filtering, individual deletion, and full-history clearing
- A completely isolated sample ticket and sample review under Demo mode
- Loading, error, empty, and not-found states
- Input limits, server-side rate limiting, timeouts, safe error mapping, and security headers
- No database and no authentication

## Architecture

```text
Browser
  ├── /simulate
  │     └── POST /api/tickets ──> OpenAI Responses API
  ├── /session/[id]
  │     └── POST /api/reviews ──> OpenAI Responses API
  ├── /history ──> versioned localStorage envelope
  └── /demo ──> static sample data, no API requests
```

The OpenAI client is created only inside `src/lib/openai.server.ts`. Client components call same-origin route handlers and never receive the API key.

## Local setup

Requirements:

- Node.js 20 or newer
- npm
- An OpenAI API key with access to `gpt-5.6`

Install dependencies:

```bash
npm install
```

Copy the environment template:

```bash
copy .env.example .env.local
```

Set the server-only variable in `.env.local`:

```dotenv
OPENAI_API_KEY=your_api_key_here
```

Never prefix this value with `NEXT_PUBLIC_`.

Start the application:

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## API

### `POST /api/tickets`

Accepts a bounded developer profile:

```json
{
  "role": "Front-End",
  "experience": "6–12 months",
  "technologies": ["React", "TypeScript"],
  "availableTime": "2 hours",
  "language": "English",
  "projectDescription": "A project management dashboard for small teams."
}
```

Returns `{ "ticket": ... }` with a validated ticket ID, context, requirements, acceptance criteria, difficulty, likely files, hint, and common mistakes.

### `POST /api/reviews`

Accepts the generated ticket plus the user's approach, code or pseudocode, difficulties, and senior question.

Returns `{ "review": ... }` with a score from 0–100, specific assessments, bugs, security concerns, improvements, educational guidance, a concise ideal solution, and skills to study.

Both routes:

- use `gpt-5.6` with medium reasoning effort;
- use Structured Outputs through `zodTextFormat`;
- validate browser input and model output;
- cap request size and model output;
- disable SDK retries to prevent duplicate calls;
- return sanitized public errors;
- never fall back to sample data.

## History format

History is stored in `juniorflow-history` as:

```json
{
  "version": 1,
  "entries": []
}
```

Every entry contains the profile, ticket, creation date, current status, optional submission, optional review, and final score inside the review. Invalid or unknown history versions are isolated and replaced with a safe empty state.

History is local to the current browser and is not synchronized across devices.

## Security

- `OPENAI_API_KEY` is read exclusively in server-only code.
- `.env.local` is ignored by Git.
- API payloads are strict, bounded Zod objects.
- Requests larger than 20 KB are rejected.
- In-memory per-instance rate limits reduce accidental cost spikes.
- OpenAI calls use a 45-second timeout and no automatic retry.
- Provider errors and stack traces are never returned to the browser.
- AI responses and API routes are marked `no-store`.
- Generated content is rendered as React text, not raw HTML.

The in-memory limiter is intentionally lightweight for this database-free MVP. Production deployments that need strong distributed enforcement should use a managed rate-limit service.

## Deploying to Vercel

1. Import the repository into Vercel.
2. Keep the detected Next.js framework settings.
3. Add `OPENAI_API_KEY` under Project Settings → Environment Variables.
4. Deploy.
5. Verify ticket generation and review in the deployed environment.

Do not commit `.env.local` or paste an API key into Vercel build logs.

## Demo mode

`/demo` is labeled **Sample ticket / Demo mode**. It uses only static local data and never calls OpenAI. API failures display real errors and an explicit link to the demo; they never substitute demo content automatically.
